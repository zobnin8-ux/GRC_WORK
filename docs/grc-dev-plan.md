## 🔨 База — делаем сейчас (порядок = очередь задач Cursor)

### Б1. Intake endpoint — СЛЕДУЮЩИЙ
- **Цель:** входная дверь. Принять лид по HTTP, посчитать ключ, записать в `leads`, поставить `enrich`. Делает уже готовую цепочку рабочей без ручных вставок.
- **Где:** Supabase Edge Function `intake` (деплой через MCP; на Vercel перенесём позже при подключении сайта).
- **Готово когда:** POST нового лида → 200 + строка в `leads` + job `enrich`; повтор → дедуп без дублей; невалидный вход → 400.

### Б2. pg_cron — автозапуск воркера
- **Цель:** движок крутится сам по расписанию, а не от ручного вызова.
- **Где:** pg_cron + pg_net внутри Supabase (бесплатно).
- **Готово когда:** воркер вызывается по расписанию (напр. каждую минуту), очередь разбирается без ручного дёрганья.

### Б3. reconciliation
- **Цель:** ночная сверка — осиротевшие сделки, лиды без касания, залипшие job.
- **Где:** Edge Function `reconcile` + cron. Запросы — из `grc-reliability-layer.md` §8.
- **Готово когда:** залипший `processing` >10 мин → `pending`; лид без касания >30 мин → запись в `reconciliation_log`.

### Б4. health-check (каркас)
- **Цель:** пинг внешних сервисов + запись в `health_checks`. Пока пингует доступное (Pipedrive, сам Supabase); Vapi/SMTP добавятся при подключении.
- **Готово когда:** строки появляются по расписанию; падение сервиса фиксируется.

### Б5. RPC `dash_*` + дашборд
- **Цель:** 9 функций под виджеты (`grc-dashboard-screen-1.md`) + фронт `GRCDashboard.jsx` на реальных данных.
- **Готово когда:** `supabase.rpc('dash_*')` возвращают нужные шейпы; экран показывает живые данные.

### Б6. Telegram-алерты
- **Цель:** единый отправитель (`lib/alert.ts`); тот же порог, что красит дашборд.
- **Готово когда:** error_rate >10%, новый dead job или падение сервиса → сообщение в чат.
